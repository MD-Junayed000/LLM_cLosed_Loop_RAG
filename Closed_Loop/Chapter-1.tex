\section{The Hallucination Problem in RAG}
\label{sec:hallucination_problem}

\begin{figure}[!htb]
    \centering
    \includegraphics[width=0.85\textwidth]{Images/Figure1.png}
    \caption[Conceptual comparison between standard open-loop RAG and the proposed closed-loop RAG pipeline.]{Conceptual comparison between standard open-loop RAG and the proposed closed-loop RAG pipeline. In standard RAG, the generated answer is returned without an internal reliability check. In the proposed system, internal activation probes estimate hallucination risk and route the answer through a controller.}
    \label{fig:open_vs_closed}
\end{figure}

Figure~\ref{fig:open_vs_closed} summarizes the motivation for this work. A standard RAG pipeline retrieves context, generates an answer, and returns the answer without verifying whether the generation is actually grounded in the retrieved evidence. This creates an \textbf{open-loop failure mode}: hallucinated content can reach the user even when the system has access to relevant documents.

The proposed pipeline adds an internal feedback stage. During generation, forward hooks extract activation features from the LLM, and lightweight probes convert those features into a hallucination-risk estimate. The controller then uses this estimate to decide whether to accept the answer, retry generation, perform re-retrieval, or abstain. The probability shown in the figure is illustrative and represents the type of signal used by the controller, not a universal threshold.

This study therefore focuses on \emph{closing the gap between hallucination detection and hallucination mitigation}: rather than only scoring a response after generation, the system uses the score to control the inference-time RAG loop.

\section{Why Hooks?}
\label{sec:why_hooks}

Before describing the full closed-loop system, it is necessary to explain how the model's internal states are accessed. This work uses \textbf{PyTorch forward hooks}, which allow selected intermediate tensors to be observed during a model's forward pass without changing the model's computation.

\subsection{What Is a Forward Hook?}
\label{subsec:forward_hook}

A forward hook is a callback function attached to a neural-network module. When the module is executed, the hook receives the module input or output and can store a copy for analysis. In this study, hooks are used \emph{passively}: they record detached activation tensors during inference and do not modify the model output, decoding process, or parameter values.

This makes hooks suitable for internal-state probing. Rather than relying only on generated text or token probabilities, the system can inspect representations formed inside the decoder while the model processes the retrieved context and generates an answer.

    \centering
    \includegraphics[width=0.70\textwidth]{Images/Figure2_Hook.png}

A hook copies an intermediate activation from a selected decoder layer while the forward pass continues unchanged. The copied activation is passed to a lightweight probe, which estimates hallucination risk for the closed-loop controller.}

\end{figure}

\subsection{Why Hooks Are Needed in This System}
\label{subsec:hooks_needed}

The proposed system requires two complementary internal signals:

\begin{enumerate}
    \item \textbf{Contextualized Embedding Vector (CEV):} the residual-stream output of a decoder block, representing the accumulated contextual state after attention and feed-forward processing.
    \item \textbf{Intermediate Activation Vector (IAV):} the SwiGLU feed-forward intermediate activation before the final $W_{\text{down}}$ projection, representing the FFN signal before it is written back into the residual stream.
\end{enumerate}

These two tensors are not equally accessible through standard model outputs. Residual hidden states can usually be returned with \texttt{output\_hidden\_states=True}, but the intermediate SwiGLU activation inside the feed-forward sublayer is not exposed by default. Forward hooks are therefore required to capture the IAV signal directly.

In the experiments, a single forward hook is registered at one mid-depth decoder layer, approximately $\ell^* = \lfloor L/2 \rfloor$. The CEV and IAV features from that layer are scored by lightweight MLP probes, and the resulting hallucination-risk estimate is passed to the closed-loop controller. Using one mid-depth layer rather than concatenating several depths is justified empirically by the layer-wise ablation in Section~\ref{sec:layer_selection}.

\subsection{Why Not Just Read Hidden States After Inference?}
\label{subsec:why_not_after}

Standard HuggingFace APIs expose only final-layer hidden states or optionally all residual-stream states. They do \emph{not} expose intermediate FFN activations (the IAV signal) without custom code. Forward hooks provide a clean, non-invasive mechanism to capture \emph{any} intermediate tensor---including the SwiGLU gated intermediate---during the live forward pass. This is why hooks are essential for this dual-signal probing architecture.
