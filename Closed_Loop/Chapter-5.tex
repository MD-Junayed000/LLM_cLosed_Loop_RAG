\chapter{Methodology}
\label{ch:methodology}

This chapter connects the pieces together. In Section~\ref{sec:background} we already established what signals we want (CEV and IAV) and why. Here we describe how such signals are integrated into a working system one that retrieves generates probes decides and sometimes retries. The chapter narrows down from high-level overview to each functional blocks, the mathematical formalization of feature extraction, the probe architecture, probe training and calibration, cross-domain transfer analysis, and the closed-loop policy controller.

%% -------------------------------------------------
\section{Overview of the Closed-Loop RAG Pipeline}
\label{sec:pipeline_overview}

The overall picture is straightforward (see Figure~\ref{fig:system_overview}). A user query arrives in. The retriever pulls the relevant passages from an FAISS index . Frozen LLM backbone generates a candidate answer, while internal activation are silently captured by forward hooks. These activations are scored by two lightweight probes, one for CEV, one for IAV, and the resulting calibrated probabilities are combined into one hallucination risk estimate. Finally, a rule-based controller receives that estimate plus a retrieval-quality score, and then determines what to do: deliver the answer, retry, fetch better evidence, or politely decline.

\section{Core Functional Blocks}
\label{sec:core_blocks}

The system is divided into four major blocks. Let's take them one at a time.
\newpage
\subsection{Retrieval Engine}
\label{subsec:retrieval_engine}

We use a standard dense-retrieval setup \cite{karpukhin2020dpr}. We embed documents with the BAAI/bge-base-en-v1.5 embedding model~\cite{xiao2023bge}, which outputs 768-dimensional vectors. These are indexed using FAISS \cite{johnson2019faiss}. At query time, we encode the user query with the same embedding model and retrieve the top-k similar passages.

We also compute a retrieval quality score -- simply defined, how good a match did we get?

\begin{equation}
    q_{\text{ret}} = \frac{1}{k} \sum_{i=1}^{k} \text{sim}(\mathbf{q},\, \mathbf{d}_i)
    \label{eq:qret}
\end{equation}

\noindent This is the mean cosine similarity between the query embedding $\mathbf{q}$ and each retrieved passage embedding $\mathbf{d}_i$. The controller uses it later as a rough proxy for whether the retrieved evidence is actually relevant enough to answer the question.

\subsection{LLM Backbone with Internal Hooks}
\label{subsec:llm_backbone}

The retrieved context and user query get assembled into a prompt and fed to a frozen decoder-only LLM. We test three backbones:
\begin{itemize}
    \item Mistral-7B-Instruct-v0.2
    \item LLaMA-3-8B-Instruct
    \item Qwen3-8B
\end{itemize}

``Frozen'' means we never touch the model weights, the backbone generates exactly as it normally would. The only addition is a forward hook registered at a single mid-depth layer (roughly $L/2$). That hook captures two things: the residual-stream output $\mathbf{z}_{\text{CEV}}^{(\ell^*)}$ and the SwiGLU intermediate $\mathbf{z}_{\text{IAV}}^{(\ell^*)}$. Why one layer rather than three? Section~\ref{sec:layer_selection} has the empirical answer.

\subsection{Hallucination Detection Network}
\label{subsec:detection_network}

Two small MLP probes operate independently:
\begin{enumerate}
    \item The \textbf{CEV probe} takes in the residual-stream feature vector $\boldsymbol{\phi}(\mathbf{x})$.
    \item The \textbf{IAV probe} takes in the FFN-intermediate feature vector $\boldsymbol{\psi}(\mathbf{x})$.
\end{enumerate}

Each probe outputs a hallucination probability:
\begin{align}
    p_{\text{CEV}} &= P(y = 1 \mid \boldsymbol{\phi}(\mathbf{x})) \label{eq:pcev}\\[4pt]
    p_{\text{IAV}} &= P(y = 1 \mid \boldsymbol{\psi}(\mathbf{x})) \label{eq:piav}
\end{align}

\noindent These are then combined through a validation-tuned convex combination; our central fusion equation:

\begin{equation}
    \boxed{\;p_{\text{fused}} = w \cdot p_{\text{CEV}} + (1 - w) \cdot p_{\text{IAV}}\;}
    \label{eq:pfused}
\end{equation}

We choose the weight $w$ on the validation split that maximizes AUROC. Interestingly, the optimal weights are not the same across models: Mistral lands at an even $0.50/0.50$ split, Qwen leans slightly towards CEV ($0.60/0.40$), and LLaMA tilts marginally towards IAV ($0.45/0.55$). The takeaway is that both signals are meaningful irrespective of backbone, neither one dominates across the board.

\subsection{Closed-Loop Policy Controller}
\label{subsec:controller_overview}

The controller makes the decisions. It takes as input $p_{\text{fused}}$, the retrieval quality score $q_{\text{ret}}$, and the current retry count. It selects one of four actions: accept, regenerate, re-retrieve, and abstain. The full decision logic is described in Section~\ref{sec:controller_logic}, but Table~\ref{tab:system_components} gives an overview of the connectivity of all components.

\section{Internal-State Feature Extraction}
\label{sec:feature_extraction}

Now for the math. Let $L$ be the total layer count. We pick our hook layer at:

\begin{equation}
    \ell^* = \lfloor L/2 \rfloor
    \label{eq:layer_selection}
\end{equation}

\noindent At that layer, we extract the CEV directly from the block output:

\begin{equation}
    \mathbf{z}_{\text{CEV}}^{(\ell^*)} = \mathbf{h}_{\text{out}}^{(\ell^*)}
    \label{eq:cev_extract}
\end{equation}

\noindent And the IAV from the gated SwiGLU intermediate:

\begin{equation}
    \mathbf{z}_{\text{IAV}}^{(\ell^*)} = \text{SiLU}\bigl(W_{\text{gate}}^{(\ell^*)} \text{RMSNorm}(\mathbf{h}_{\ell^*})\bigr) \odot \bigl(W_{\text{up}}^{(\ell^*)} \text{RMSNorm}(\mathbf{h}_{\ell^*})\bigr)
    \label{eq:iav_extract}
\end{equation}

Since we are working with autoregressive models, we use the \emph{last token's} representation---it is the only position that has attended to the complete preceding context and generated text:

\begin{align}
    \boldsymbol{\phi}(\mathbf{x}) &= \mathbf{z}_{\text{CEV,last}}^{(\ell^*)} \label{eq:phi}\\[3pt]
    \boldsymbol{\psi}(\mathbf{x}) &= \mathbf{z}_{\text{IAV,last}}^{(\ell^*)} \label{eq:psi}
\end{align}

\noindent Table~\ref{tab:hook_layers} shows exactly which layer we hook for each backbone and what dimensions result.

\section{Probe Architecture and Calibration}
\label{sec:probe_arch}

Each hallucination probe is deliberately kept small with a two-hidden-layer MLP. We use separate probes for CEV and IAV because these features live in different vector spaces (4,096-dim vs. 12,288--14,336-dim) and represent fundamentally different computational signals.

The architecture, written out as a pipeline:
\[
\begin{aligned}
\text{Input} &\to \text{Linear}(d, 256) \to \text{BN} \to \text{ReLU} \to \text{Dropout} \\
&\to \text{Linear}(256, 128) \to \text{BN} \to \text{ReLU} \to \text{Dropout} \to \text{Linear}(128, 2)
\end{aligned}
\]

After training, we apply temperature scaling to the output logits before computing the softmax:

\begin{equation}
    P(y = 1) = \text{softmax}\!\left(\frac{\mathbf{z}}{T}\right)
    \label{eq:temp_scaling}
\end{equation}

\noindent The temperature $T$ is tuned on the validation split by minimizing negative log-likelihood. This calibration step is important because the controller needs well-calibrated probabilities, not just good rankings it makes threshold-based decisions, so the absolute probability values matter.

\section{Training and Fusion Summary}
\label{sec:training_fusion_summary}

The probes are trained on labeled RAG hallucination examples (from RAGTruth) using class-weighted cross-entropy with label smoothing. We optimize with AdamW~\cite{loshchilov2019adamw,kingma2015adam} and apply temperature scaling for post-training calibration~\cite{guo2017calibration}. The two probes are trained independently, they never see each other's features and fused only after calibration via Eq.~\eqref{eq:pfused}.

Full training details (optimizer settings, learning rates, early stopping, etc.) are consolidated in Table~\ref{tab:training_config} in Section~\ref{sec:training_protocol}.

\section{Example Closed-Loop Trace}
\label{sec:example_trace_ch5}

To make the system concrete, here are two worked examples showing what happens end-to-end. The numbers are illustrative, they show the kind of computation the controller performs, not exact logged values.

\begin{groundedbox}[Example A: Grounded Response $\to$ ACCEPT]
\begin{tabular}{@{}r@{\hspace{8pt}}p{10.5cm}@{}}
\textbf{Query}      & ``What year did the Treaty of Westphalia end the Thirty Years' War?'' \\[3pt]
\textbf{Retrieved}  & Passage states the Peace of Westphalia was signed in 1648 \\[3pt]
\textbf{Response}   & ``The Treaty of Westphalia ended the Thirty Years' War in 1648.'' \\[3pt]
\textbf{$q_{\text{ret}}$} & 0.72 \\[2pt]
\textbf{$p_{\text{CEV}}$} & 0.24 \quad\quad \textbf{$p_{\text{IAV}}$} = 0.28 \\[2pt]
\textbf{$p_{\text{fused}}$} & 0.27 $<$ $\theta_{\text{hallu}}$ = 0.62 \\[3pt]
\textbf{Decision}   & \textcolor{acceptgreen}{\textbf{ACCEPT}} --- Response delivered to user \\
\end{tabular}
\end{groundedbox}

\vspace{0.3em}
\noindent Both probes agree the response is low-risk. The fused score sits well below the intervention threshold. The controller accepts without hesitation.

\begin{hallucinatedbox}[Example B: Unsupported Response $\to$ ABSTAIN]
\begin{tabular}{@{}r@{\hspace{8pt}}p{10.5cm}@{}}
\textbf{Query}      & ``What is the official 2024 census population of Atlantis?'' \\[3pt]
\textbf{Retrieved}  & Passage discusses South Atlantic territories (no mention of Atlantis) \\[3pt]
\textbf{Response}   & ``According to the 2024 census, Atlantis had a population of 140,634.'' \\[3pt]
\textbf{$q_{\text{ret}}$} & 0.26 \\[2pt]
\textbf{$p_{\text{CEV}}$} & 0.71 \quad\quad \textbf{$p_{\text{IAV}}$} = 0.84 \\[2pt]
\textbf{$p_{\text{fused}}$} & 0.80 $\geq$ $\theta_{\text{abs}}$ = 0.75 \\[3pt]
\textbf{Decision}   & \textcolor{abstainred}{\textbf{ABSTAIN}} --- ``I cannot verify this claim.'' \\
\end{tabular}
\end{hallucinatedbox}

\vspace{0.3em}
\noindent Here both probes fire strongly, the CEV shows poor context coherence, and the IAV shows confabulation patterns. The fused score exceeds the abstention threshold, so the system refuses rather than delivering a fabricated number.

\section{Probe Training, Calibration, and Cross-Domain Transfer}
\label{sec:probe_training}

Having described the architecture, we now turn to how the probes are actually trained, how we ensure their outputs are well-calibrated, and critically whether the learned signals hold up when tested on data the probes have never seen before.

%% -------------------------------------------------
\subsection{Probe Architecture and Training Protocol}
\label{sec:training_protocol}

For each backbone, we train two distinct MLP probes, one taking the CEV feature vector $\boldsymbol{\phi}(\mathbf{x})$ and one taking the IAV feature vector $\boldsymbol{\psi}(\mathbf{x})$. Both employ the same two-hidden-layer architecture as explained in Section~\ref{sec:probe_arch} (Table~\ref{tab:probe_architecture}) but use different input dimensions as CEV and IAV originate from separate representational spaces.

The labeling convention is straightforward: $y = 0$ signifies that the response is grounded (supported by the retrieved context), while $y = 1$ means that it is hallucinated. After training converges, we convert the raw logits into calibrated probabilities via temperature scaling, a very important step when the downstream controller makes decisions based on absolute probability values instead of just ranks.

Table~\ref{tab:training_config} provides a summary of the complete training procedure.

\newpage
\subsubsection{In-Domain Training Behavior}
\label{sec:indomain_training}

Training is well-behaved across the board. CEV and IAV probe loss curves drop smoothly throughout all three backbones; no instability, no unexpected collapses. Validation AUROC increases then plateaus which tells us the probes are learning real patterns instead of memorizing training instances.

And it is especially intriguing because the dual-signal architecture is empirically valid. Although IAV is usually the strongest individual branch, CEV is seldom far behind and definitely adds additional information. This motivates our approach of training two separate probes and combining their outputs, rather than using a single source of signal.

\subsubsection{HaluEval Cross-Domain Transfer}
\label{sec:halueval_transfer}

The bottom row of Figure~\ref{fig:training_halueval} shows an interesting case.

For LLaMA-3-8B-Instruct and Qwen3-8B, transfer works as expected. The raw AUROC scores are greater than 0.5, indicating that the probes trained on RAGTruth still give larger hallucination scores to the hallucinated HaluEval responses than to the right ones. The signal is generalizable.

Mistral-7B-Instruct-v0.2 is a different narrative Its raw AUROC values are well \emph{below} 0.5, on the order of 0.105. At first sight this appears like a complete failure. But look a little closer, an AUROC of 0.105 is not random (that would be 0.5). It means that the probe is separating the classes \emph{really well} in the opposite direction. The adjustment $1 - 0.105 = 0.895$ shows that the discriminative power is actually excellent, simply the polarity flipped under distribution shift.

The complete numerical breakdown is available in Table~\ref{tab:halueval_ood} (Section~\ref{sec:halueval_ood}). 

\subsubsection{Interpretation of Mistral Polarity Inversion}
\label{sec:polarity_inversion}

It is appropriate to interpret this polarity inversion that it is \emph{not} a code bug and if it was random noise, the AUROC would be around 0.5, not plummet to 0.105. And this is not because we utilize a base model instead of an instruction tuned one; we expressly use the fully tuned Mistral-7B-Instruct-v0.2 checkpoint.

What seems to be happening is that Mistral's internal representations encode hallucination-relevant information in a fashion that is directionally particular to the training distribution. When the distribution changes (from RAGTruth to HaluEval) the \emph{meaning} of some activation patterns flips. The signal is strong yet its sign is inverted.

The practical message is sobering, but important: \emph{you cannot trust that an internal-state probe will carry over automatically to a different domain, benchmark, or prompt format.} Before being applied in a new situation, the probe should be validated and potentially re-calibrated on representative instances from the target distribution.

\subsubsection{Comparison with the Triple-Layer Configuration}
\label{sec:triple_comparison}

One might question whether our single mid-layer approach is better than a three-layer concatenation at dealing with distribution shift; perhaps the additional depth diversity can be useful for generalization? We examined. It does not.

\subsection{Post-Training Calibration and Fusion}
\label{sec:calibration_fusion}

Once training is complete, we calibrate each probe independently using temperature scaling:
\begin{equation}
    P(y = 1) = \text{softmax}\!\left(\frac{\mathbf{z}}{T}\right)
\end{equation}

\noindent  The temperature $T$ is learned via minimization of validation negative log-likelihood. Why do we have to do this anyway? Since the controller makes decisions based on thresholds. If the probe gives 0.65 when it should be 0.45, the controller will overcontrol. Calibration ensures that the probability values \emph{actually mean what they do}.

We calibrate CEV and IAV separately because the two probes can have quite different confidence profiles. One might be well-ranked but systematically overconfident; the other might be better calibrated but with weaker discrimination. Temperature scaling brings them onto a common scale before fusion.

After calibration, the two probabilities are combined:
\[
    p_{\text{fused}} = w \cdot p_{\text{CEV}} + (1 - w) \cdot p_{\text{IAV}}
\]

\noindent with $w$ selected by grid search on the validation split to maximize AUROC. Importantly, this fusion happens at the \emph{probability} level---not by concatenating raw activation vectors into a bigger classifier. Each probe keeps its own learned decision boundary; we combine their judgments, not their inputs.

\begin{table}[!htb]
\centering
\caption{Calibration and fusion settings.}
\label{tab:calibration_settings}
\begin{tabular}{@{}ll@{}}
\toprule
\textbf{Component} & \textbf{Setting} \\
\midrule
Calibration method        & Temperature scaling \\
Calibration scope         & Applied separately to CEV and IAV probes \\
Calibration objective     & Minimize validation NLL \\
Fusion type               & Probability-level convex fusion \\
Fusion formula            & $p_{\text{fused}} = w \cdot p_{\text{CEV}} + (1-w) \cdot p_{\text{IAV}}$ \\
Fusion weight selection   & Validation-set grid search \\
Fusion objective          & Maximize validation AUROC \\
Controller input          & $p_{\text{fused}}$ \\
\bottomrule
\end{tabular}
\end{table}

\newpage
\subsection{Probe Training Summary}
\label{sec:ch6_summary}

What have we learned in this section? Three things stand out.

First, the probes train stably and learn true hallucination-relevant patterns on RAGTruth both CEV and IAV converge smoothly, and validation metrics confirm generalization to the held-out split.

Second, the out-of-domain transfer is real but model dependent. Qwen3-8B and LLaMA-3-8B-Instruct still perform well in their discriminative manner on HaluEval. Mistral-7B-Instruct-v0.2 is still very discriminative, but the polarity is reversed, an interesting result showing how sensitive these probes can be to the distribution.

Third, calibration is not an option. The downstream controller makes absolute threshold decisions, so the probes need to produce probabilities associated with empirical risk levels. This is achieved with temperature scaling and probability-level fusion, which keeps the two signals separate but combines their strengths.

The bottom line is that internal-state probes are effective, but they are not plug-and-play across domains. Deploy with confidence. Validate first.

\section{Closed-Loop Policy Controller}
\label{sec:controller}

Up to this point, we have a system that can detect hallucination risk but detecting is not the same as \emph{doing something about it}. This section is about the ``doing'' part. We describe how the fused hallucination probability gets turned into an actual operational decision: should the response go to the user, should the system try again, or should it just say ``I don't know''?

%% -------------------------------------------------
\subsection{Purpose of the Controller}
\label{sec:controller_purpose}

The preceding sections built up a detection pipeline: extract CEV and IAV features, score them with calibrated probes, fuse the results into $p_{\text{fused}}$. But a number on its own does not help the user. What matters is what the system \emph{does} with that number.

That is the controller's job. It takes the hallucination-risk estimate and converts it into an inference-time action. Not a label, not a flag in a log file---an action that changes what happens next in the pipeline.

The controller receives three inputs:
\begin{itemize}
    \item $p_{\text{fused}}$---the calibrated fused hallucination-risk probability
    \item $q_{\text{ret}}$---the retrieval-quality score, computed as the mean similarity of the top-$k$ retrieved passages
    \item $r$---the current retry count (how many times we have already looped)
\end{itemize}

And it picks one of four actions:
\[
    \mathcal{A} = \{\textsc{Accept},\; \textsc{Regenerate},\; \textsc{Re-Retrieve},\; \textsc{Abstain}\}
\]

%% -------------------------------------------------
\subsection{Controller Inputs and Thresholds}
\label{sec:controller_inputs}

Three thresholds govern the controller's behavior. Each one answers a different question.

\begin{table}[!htb]
\centering
\caption{Controller threshold symbols and their roles.}
\label{tab:threshold_symbols}
\begin{tabular}{@{}lll@{}}
\toprule
\textbf{Symbol} & \textbf{Meaning} & \textbf{Question It Answers} \\
\midrule
$\theta_{\text{hallu}}$ & Intervention threshold & Is the response safe enough to deliver? \\
$\theta_{\text{abs}}$   & Abstention threshold  & Is the response so risky we should give up? \\
$\theta_{\text{ret}}$   & Retrieval-quality threshold & Was the retrieved evidence good enough? \\
\bottomrule
\end{tabular}
\end{table}

Two of these are fixed across all backbones in our experiments: $\theta_{\text{ret}} = 0.30$ and $\text{MAX\_RETRIES} = 3$. The hallucination and abstention thresholds vary per backbone because each model's activation distribution produces different probe-score ranges (see Table~\ref{tab:controller_config}).

%% -------------------------------------------------
\subsection{Controller Decision Logic}
\label{sec:controller_logic}

The policy is entirely deterministic---no learned weights, no stochastic sampling, no neural network in the loop. Given $p_{\text{fused}}$, $q_{\text{ret}}$, and retry count $r$, the controller follows this rule:

\begin{equation}
\text{Action} = 
\begin{cases}
    \textsc{Abstain}      & \text{if } r \geq \text{MAX\_RETRIES} \\[4pt]
    \textsc{Accept}       & \text{if } p_{\text{fused}} < \theta_{\text{hallu}} \\[4pt]
    \textsc{Abstain}      & \text{if } p_{\text{fused}} \geq \theta_{\text{abs}} \\[4pt]
    \textsc{Re-Retrieve}  & \text{if } \theta_{\text{hallu}} \leq p_{\text{fused}} < \theta_{\text{abs}} \;\wedge\; q_{\text{ret}} < \theta_{\text{ret}} \\[4pt]
    \textsc{Regenerate}   & \text{if } \theta_{\text{hallu}} \leq p_{\text{fused}} < \theta_{\text{abs}} \;\wedge\; q_{\text{ret}} \geq \theta_{\text{ret}}
\end{cases}
\label{eq:controller_rule}
\end{equation}

The logic is read top to bottom---first check wins. The interesting part is what happens in the middle zone, when the risk is elevated but not extreme. There, the controller asks a follow-up question: \emph{was the retrieval any good?} If the evidence looks weak ($q_{\text{ret}}$ below threshold), the problem is probably upstream bad retrieval so the system fetches new evidence. If the evidence was fine but the answer is still risky, the problem is likely in generation itself, so the system regenerates with the same context.

%% -------------------------------------------------
\subsection{Semantic Meaning of Each Controller Action}
\label{sec:action_semantics}

Each action carries a distinct meaning about where the system thinks the problem lies.

\begin{table}[!htb]
\centering
\caption{What each action means and what the system does.}
\label{tab:action_semantics}
\small
\begin{tabular}{@{}l p{3cm} p{4cm} p{4cm}@{}}
\toprule
\textbf{Action} & \textbf{When It Fires} & \textbf{What It Means} & \textbf{What Happens} \\
\midrule
\textbf{ACCEPT} & $p_{\text{fused}} < \theta_{\text{hallu}}$ & ``Looks safe'' & Deliver the response to the user \\[4pt]
\textbf{REGENERATE} & Moderate risk, good retrieval & ``Evidence is fine, generation isn't'' & Generate a new answer using same context \\[4pt]
\textbf{RE-RETRIEVE} & Moderate risk, weak retrieval & ``The evidence was the problem'' & Fetch better passages, then try again \\[4pt]
\textbf{ABSTAIN} & Very high risk or out of retries & ``Can't safely answer this'' & Return a polite refusal \\
\bottomrule
\end{tabular}
\end{table}

The key distinction worth emphasizing: REGENERATE blames the generator, while RE-RETRIEVE blames the retriever. This separation lets the system target its corrective effort where it is most likely to help.

\label{sec:trace_ch7}

To make the controller's behavior tangible, consider the following illustrative trace. We ask about the location of Atlantis---a query designed to invite a confident hallucination, since the premise is fictional.

A few things to notice here. First, the controller did not simply refuse on the first attempt---it tried to recover. The first response was risky \emph{and} the retrieval quality was poor, so it fetched new evidence. The second time, retrieval was better but the generation was still speculative, so it regenerated. Only on the third try did the model produce something safe enough to deliver.

Second, the final accepted answer is itself a kind of refusal (``Atlantis is mythological; I cannot verify a single factual location''). But from the controller's perspective, this is not an ABSTAIN---it is an ACCEPT of a response that happens to be cautious. The system accepted a low-risk output; it did not withhold a response. There is an important semantic distinction there.

\textbf{Note on the real-time executions.} This trace is an illustration, built to depict the complete recovery process. In the actual nine-query demo (Section~\ref{sec:nine_query_demo}), the true Atlantis query worked in one try on all three backbones. Qwen3-8B refused outright ($p_{\text{fused}} = 0.80$), but Mistral and LLaMA both accepted the model's own in-text denial on the first try ($p_{\text{fused}} = 0.31$ and $0.10$ respectively). This multi-round recovery path shows what \emph{would} happen if the initial attempt landed in the intervention zone---it was not triggered by this particular query in practice.

\subsection{Controller Summary}
\label{sec:ch7_summery}

The controller is deliberately kept simple, a handful of threshold comparisons and an if-else chain.  That simplicity is a strength, not a weakness. This means the system is completely interpretable: given the three inputs, anyone can follow exactly why a certain action was chosen. There's no secret weights to debug, no stochastic decisions to second guess.

There is a lot to be gained from the controller, despite its simple nature. This shifts hallucination detection from a passive observation (``this response might be risky'') into an active intervention (``let's try again'' or ``let's not answer''). The RAG pipeline is self-corrective at inference time, and it does so in a way that is clear enough to audit and explain.
